## Что такое система контроля версий?

*Система контроля версий (VCS - Version Control System) — это программное обеспечение, которое помогает отслеживать изменения в файлах и координировать работу над проектом с другими разработчиками.*

Для чего нужна система контроля версий?

•	История изменений: Возможность вернуться к любой версии файла.

•	Коллаборация: Множество людей могут работать над одним проектом без конфликтов.

•	Отслеживание ошибок: Легко найти, когда и кто внес изменения, которые привели к ошибке.

•	Резервное копирование: Автоматическое создание резервных копий вашего кода.

## История Git и его основные концепции

Git — это распределённая система контроля версий, созданная Линусом Торвальдсом для разработки ядра Linux. В отличие от централизованных систем (например, SVN), каждая копия репозитория Git является полной и содержит всю историю изменений.

## Что такое репозиторий и создание локальных репозиториев

**Репозиторий**

Репозиторий — это место, где хранятся все файлы проекта, включая историю изменений.

Создание локального репозитория
1.	Откройте папку вашего проекта.
2.	В терминале или командной строке выполните: 

```sh
git init
```
________________________________________
## Базовая работа с локальным репозиторием

Просмотр состояния репозитория:

```sh
git status
```

Добавление файлов в staging area:
```sh
git add <имя-файла>
```

или:
```sh
git add .
```

для добавления всех файлов.

**Создание файла .gitignore:**

Файл .gitignore позволяет указать файлы и папки, которые Git должен игнорировать. 

Пример:
```sh
*.log
node_modules/
.DS_Store
```

Создайте файл в корне проекта и добавьте туда ненужные файлы/папки.

**Создание коммита:**

```sh
git commit -m "Краткое и информативное сообщение о коммите"
```

**Просмотр истории коммитов:**

```sh
git log
```
или:
```sh
git log --oneline
```

для краткого отображения.

## Что такое ветки и для чего они нужны

**Ветки позволяют работать над различными версиями одного проекта параллельно.**

Зачем нужны ветки:

•	Изоляция разработки новых функций.

•	Исправление ошибок без влияния на основную версию.

## Базовая работа с ветками в Git

**Создание ветки:**

```sh
git branch <имя-ветки>
```

**Переключение на ветку:**

```sh
git checkout <имя-ветки>
```

или в одну команду: 

```sh
git checkout -b <имя-ветки>
```

**Удаление ветки:**

```sh
git branch -d <имя-ветки>
```

**Слияние веток:**

```sh
git merge <имя-ветки>
Rebase (перебазирование): 
git rebase <имя-ветки>
```

### Альтернатива слиянию, создающая более "чистую" историю.

**Визуализация веток:**

Для наглядного отображения дерева веток используйте:

```sh
git log --oneline --graph --all
```

**Стратегии ветвления:**

•	Gitflow: Подходит для сложных проектов с длительными циклами разработки.

•	GitHub Flow: Простая стратегия для непрерывной интеграции.

•	Feature Branching: Каждая новая функция разрабатывается в отдельной ветке.

## Что такое удалённый репозиторий и для чего он нужен

**Удалённый репозиторий**

*Удалённый репозиторий — это репозиторий, размещённый на сервере (например, GitHub), позволяющий синхронизировать ваш локальный репозиторий с сервером для совместной работы.*

Базовая работа с удалёнными репозиториями

•	Добавление удалённого репозитория:
```sh 
git remote add origin <URL-репозитория>
```

•	Отправка изменений: 
```sh
git push -u origin <имя-ветки>
```

•	Загрузка изменений: 
```sh
git pull origin <имя-ветки>
```

•	Клонирование репозитория: 
```sh
git clone <URL-репозитория>
```

•	Получение изменений без слияния: 
```sh
git fetch origin
```

## Совместная работа в Git

Инструкция по созданию Pull Request

1.	Загрузите вашу ветку на GitHub с изменениями: 
2.	git push origin <имя-ветки>
3.	Перейдите на GitHub, в ваш репозиторий.
4.	Нажмите на вкладку Pull requests, затем New pull request.
5.	Выберите, из какой ветки в какую нужно слить изменения.
6.	Добавьте описание и создайте PR.

**Работа с Pull Request:**

* Обсуждение: Участники проекта могут комментировать изменения.

* Code Review: Просмотр и обсуждение кода другими разработчиками.

* Разрешение конфликтов: Git укажет на конфликтующие участки, которые нужно исправить вручную.

* Слияние PR: 

* Merge: Стандартное слияние.

* Squash: Объединяет все изменения в один коммит.

* Rebase: Перебазирование изменений для сохранения линейной истории.

**Работа с задачами (Issues):**
* Используются для отслеживания ошибок, новых функций и предложений.
* Задачи связываются с Pull Requests для удобного отслеживания.

## Решение часто встречающихся проблем

1.	Ошибка "fatal: not a git repository":

Проверьте, инициализирован ли репозиторий (git init).

2.	Конфликты при слиянии:

Git укажет конфликтующие файлы. Ручной способ: 
Откройте файл с конфликтами, исправьте их вручную.

Добавьте файл в staging area (git add <имя-файла>).

Завершите слияние: 
```sh
git commit
```

3.	Ошибка при push ("Updates were rejected"):

Выполните git pull перед отправкой: 
```sh
git pull origin <имя-ветки>
```
затем снова выполните git push.

## Альтернативные системы контроля версий

* Mercurial: Похожа на Git, но с более простым синтаксисом.
* Subversion (SVN): Централизованная система, часто используется в старых проектах.
* Perforce: Подходит для крупных корпоративных команд и больших файлов.
