# Инструкция по работе с git

## Что это и для чего нужна система контроля версий?
Системы контроля версий (VCS) являются критически важными инструментами для управления и отслеживания изменений в файлах и проектах, особенно в сфере разработки.

### Что такое система контроля версий?
Инструмент, который дает возможность видеть, кто, когда и какие изменения вносил в проект; позволяет формировать новые ветви проекта, объединять уже имеющиеся; настраивать контроль доступа к проекту; осуществлять откат до предыдущих версий.
### Для чего нужна система контроля версий
#### 1. Отслеживание Изменений
Системы контроля версий позволяют отслеживать все изменения, внесенные в файлы или код, включая информацию о том, кто сделал изменения, когда они были сделаны, и почему.
#### 2. Возвращение к Предыдущим Версиям
VCS позволяет вернуться к любой предыдущей версии файла или проекта, если необходим откат из-за ошибок или других проблем. Это особенно полезно при исправлении багов или восстановлении работоспособной версии проекта.
#### 3. Коллаборация и Управление Командой
Для команд разработчиков VCS обеспечивает возможность работать над одним проектом одновременно, без риска конфликтов между изменениями. Системы контроля версий помогают управлять доступом к проекту, формировать новые ветви разработки и объединять существующие.
#### 4. Безопасность и Резервное Копирование
Распределенные системы контроля версий, такие как Git, обеспечивают полное резервное копирование всех данных проекта на каждом клиентском репозитории. Это означает, что даже если центральный сервер выйдет из строя, любой клиентский репозиторий может быть использован для восстановления проекта.
#### 5. Управление Версиями и История Изменений
VCS сохраняет всю историю изменений, что позволяет разработчикам анализировать первопричины возникновения ошибок и понимать, почему код структурирован определенным образом. Это особенно важно для работы с унаследованным кодом.
#### 6. Увеличение Продуктивности
Использование VCS помогает командам разработчиков работать быстрее и более эффективно, сокращая время разработки и увеличивая количество успешных развертываний. Это также помогает в управлении сложными проектами и поддержании гибкости при росте команды.
В целом, системы контроля версий являются фундаментальным инструментом для любого проекта, требующего управления и отслеживания изменений, обеспечивая безопасность, коллаборацию и эффективность работы.
## Установка git и VSCode на ваш ПК.

### Установка VSCode на ваш ПК.
#### Шаги по установке VSCode на Windows
* **_Загрузка Установщика_**:
перейдите на официальный сайт Visual Studio Code и загрузите установщик для Windows.
* **_Запуск Установщика_**:
запустите загруженный файл и следуйте инструкциям установки. Принять условия лицензионного соглашения и нажать "Next".
* **_Выбор Директории_**:
выберите директорию, куда будет установлен VSCode. Укажите путь и нажмите "Next".
* **_Выбор Компонентов_**:
выберите компоненты, которые будут установлены. Обычно достаточно оставить компоненты по умолчанию, но можно включить дополнительные, такие как добавление в контекстное меню или создание ярлыков на рабочем столе и в меню "Пуск".
* **_Установка_**:
нажмите "Install" и дождитесь окончания установки.
### Установка git на ваш ПК
#### Шаги по установке Git на Windows
* **_Загрузка Установщика_**:
перейдите на официальный сайт Git и загрузите установщик для Windows.
* **_Запуск Установщика_**:
запустите загруженный файл и следуйте инструкциям установки. Принять условия лицензионного соглашения и нажать "Next".
* **_Выбор Директории_**:
выберите директорию, куда будет установлен Git. Укажите путь и нажмите "Next".
* **_Выбор Компонентов_**:
выберите компоненты, которые будут установлены. Обычно достаточно оставить компоненты по умолчанию, но можно включить дополнительные, такие как Git Bash или автоматическая проверка обновлений.
* **_Выбор SSL/TLS-Библиотеки_**:
выберите библиотеку для HTTPS-подключений. Обычно используется OpenSSL, но можно выбрать Windows Secure Channel если необходимо.
* **_Настройка Окончания Строк_**:
выберите вариант обработки окончания строк. Рекомендуется выбрать "Checkout Windows-style, commit Unix-style line endings" для совместимости между Windows и Unix-системами.
* **_Выбор Терминала_**:
выберите терминал для Git Bash. MinTTY является рекомендуемым вариантом.
* **_Дополнительные Настройки_**:
выберите поведение для команды git pull и включите Git Credential Manager, если необходимо. Также можно включить кэширование файловой системы и символические ссылки.
* **_Установка_**:
нажмите "Install" и дождитесь окончания установки.
#### Первая настройка git
После установки, необходимо выполнить первоначальную настройку Git:
```
git config --global user.name "Ваше Имя"

git config --global user.email "ваш.эмейл@example.com"
```
## Создание и базовая работа с локальным репозиторием.

### Что такое репозиторий и инструкция по созданию локальных репозиториев.
Репозиторий Git — это виртуальное хранилище проекта, где хранятся версии кода для доступа по мере необходимости. Это включает в себя все файлы проекта, историю их изменений, и другую служебную информацию.
Локальный Репозиторий: Хранится на локальном компьютере и используется одним разработчиком. Локальный репозиторий создается с помощью команды git init в существующем каталоге проекта.

Если у вас уже есть проект в каталоге, который не находится под версионным контролем Git, вы можете инициализировать репозиторий в этом каталоге следующим образом:
1. Перейдите в Каталог Проекта:
Используйте команду cd для перехода в каталог вашего проекта.
```
cd C:\path\to\your\project
```
2. Инициализируйте Репозиторий:
Выполните команду git init для создания нового подкаталога .git в вашем рабочем каталоге.
```
git init
```
Это создаст новую главную ветку и подготовит структуру репозитория.

### Базовая работа с локальным репозиторием
1. Добавление Файлов в Индекс
После инициализации репозитория, вам нужно добавить файлы в индекс, чтобы они были отслеживаемы Git:
```
git add <file_name>  (Для добавления конкретного файла)
```
git add .  (Для добавления всех файлов в текущем каталоге)

2. Создание Первого Коммита
Затем, зафиксируйте изменения с помощью команды git commit:
```
git commit -m "Initial commit"
```
Эта команда создаст первый коммит с указанным сообщением.
## Ветки. Локальная работа с ветками в git.

### Что такое ветки и для чего они нужны при работе с системой контроля версий.
Ветки в системе контроля версий Git являются мощным инструментом для управления и организации кода во время разработки. 

Ветки в Git представляют собой указатели на конкретные коммиты в истории проекта. Они дают возможнсть работать над разными версиями кода параллельно, не мешая друг другу.
### Базовая работа с ветками в git.
#### Создание Новой Ветки
Чтобы создать новую ветку, используйте команду git branch с именем новой ветки:
```
git branch <new-branch>
```
Эта команда создает новую ветку, но не переключает на нее. Если вы хотите сразу переключиться на новую ветку, используйте команду git checkout с флагом -b или более современный синтаксис git switch:
```
git checkout -b <new-branch>

git switch -c <new-branch>

```
#### **Переключение между Ветками**

#### Просмотр Веток
Чтобы просмотреть список всех локальных веток, используйте команду:
```
git branch
```
Эта команда покажет все ветки, и текущая ветка будет отмечена звездочкой (*).

#### Переключение на Другую Ветку
Чтобы переключиться на другую ветку, используйте команду:
```
git checkout <branch-name>
```
или более современный синтаксис:
```
git switch <branch-name>
```
Эта команда переключит ваш рабочий каталог на указанную ветку.

#### **Слияние Веток**

#### Слияние Изменений
Чтобы слить изменения из одной ветки в другую, сначала переключитесь на ту ветку, в которую вы хотите внести изменения:
```
git checkout <target-branch>
```
Затем выполните команду слияния:
```
git merge <source-branch>
```
Эта команда объединит изменения из source-branch в target-branch. Если возникнут конфликты, Git сообщит о них, и вам придется их разрешить.

#### **Управление Ветками**
#### Закоммитить Изменения в Новой Ветке
После внесения изменений в новой ветке, сохраните их с помощью команд:
```
git add .
git commit -m "<комментарий>"
```
Эти команды добавят изменения в индекс и создадут новый коммит в текущей ветке.
#### Просмотр Веток и Коммитов
Чтобы вывести список существующих веток и последний сохраненный коммит в каждой, используйте команды:
```
git branch -v
```
Эта команда покажет ветки и последние коммиты в них.
#### Переименование Ветки
Чтобы переименовать ветку, на которой вы находитесь, используйте команду:
```
git branch -m <new-name>
```
Если вам необходимо переименовать другую ветку, воспользуйтесь командой:
```
git branch -m <old-name> <new-name>
```
После переименования ветки в локальном репозитории, не забудьте удалить старую ветку из удаленного репозитория, если необходимо:
```
git push origin :<old-name>
```
#### **Удаление Веток**

#### Безопасное Удаление
Чтобы удалить локальную ветку, которая уже была слита с текущей веткой, используйте команду:
```
git branch -d <branch-name>
```
Эта команда безопасна, поскольку Git не позволит удалить ветку, если в ней есть несоединенные изменения.
#### Принудительное Удаление
Чтобы принудительно удалить ветку, даже если в ней есть несоединенные изменения, используйте команду:
```
git branch -D <branch-name>
```
Эта команда удалит ветку независимо от ее состояния.

#### **Пример Рабочего Процесса**

#### Создание Ветки для Исправления Ошибки
Если вы обнаружили критическую ошибку, которую нужно исправить, создайте новую ветку для исправления:
```
git checkout master
git checkout -b hotfix
```
После тестирования и подтверждения исправления, слейте ветку с основной:
```
git checkout master
git merge hotfix
```
Затем удалите ветку, если она больше не нужна:
```
git branch -d hotfix
```
#### Создание Ветки для Реализации Новой Функциональности
Для реализации новой функциональности, создайте новую ветку и переключитесь на нее:
```
git checkout -b feature/new-feature
```
Работайте в этой ветке, делайте коммиты и когда работа будет завершена, слейте ветку с основной:
```
git checkout master
git merge feature/new-feature
```
Затем удалите ветку, если она больше не нужна:
```
git branch -d feature/new-feature
```

Эти базовые операции с ветками в Git помогут вам эффективно управлять вашим кодом, работать над разными функциями или исправлениями параллельно, и затем сливать изменения в основную ветку.
## Работа с удаленными репозиториями.

### Что такое удаленный репозиторий и для чего он нужен
Удаленный репозиторий в системе контроля версий Git представляет собой внешний сервер или хранилище, где хранятся и обмениваются версии кода между разработчиками. 

Удаленный репозиторий — это копия вашего проекта, хранящаяся на удаленном сервере. Это может быть сервис хостинга кода như GitHub, GitLab, BitBucket или любой другой сервер, на котором установлен Git.

#### Функции Удаленного Репозитория
Хранение и Обмен Кодом
1. *__Удаленный репозиторий позволяет хранить и обмениваться кодом между разработчиками.__*

 Вы можете отправлять свои локальные изменения в удаленный репозиторий и получать изменения от других разработчиков.

2. *__Совместная Работа__*

Удаленный репозиторий необходим для совместной работы над проектом. Разработчики могут клонировать репозиторий, работать над ним локально, а затем отправлять свои изменения обратно в удаленный репозиторий. Это позволяет нескольким людям работать над одним проектом одновременно без конфликтов.

3. *__Резервное Копирование__*
Удаленный репозиторий служит как резервная копия вашего проекта. Если локальная копия проекта будет утеряна или повреждена, вы всегда можете восстановить ее из удаленного репозитория.

4. *__Управление Доступом__*
Удаленный репозиторий позволяет управлять доступом к коду. Вы можете назначать разные уровни доступа для различных пользователей, таких как чтение, запись или администрирование.
### Базовая работа с удаленными репозиториями GitHub
#### Добавление Удаленного Репозитория
Чтобы добавить новый удаленный репозиторий, используйте команду git remote add:
```
git remote add <remote-name> <remote-repo-url>
```
Например:
```
git remote add origin https://github.com/OWNER/REPOSITORY.git
```
#### Просмотр Удаленных Репозиториев

Чтобы просмотреть список текущих удаленных репозиториев, используйте команду git remote -v:
```
git remote -v
```
Эта команда покажет список удаленных репозиториев с их URL-адресами для извлечения (fetch) и отправки (push) данных.
#### Изменение URL-адреса Удаленного Репозитория
Чтобы изменить URL-адрес существующего удаленного репозитория, используйте команду git remote set-url:
```
git remote set-url `<remote-name>` `<new-repo-url>`
```
Например:
```
git remote set-url origin https://github.com/OWNER/NEW-REPOSITORY.git
```
#### Удаление Удаленного Репозитория
Чтобы удалить удаленный репозиторий, используйте команду git remote rm:
```
git remote rm <remote-name>
```
Например:
```
git remote rm origin
```
#### Примеры Использования
Клонирование Удаленного Репозитория

Чтобы клонировать существующий удаленный репозиторий, используйте команду git clone:
```
git clone `<remote-repo-url>`
```
Например:
```
git clone https://github.com/OWNER/REPOSITORY.git
```
Отправка Изменений в Удаленный Репозиторий

Чтобы отправить локальные изменения в удаленный репозиторий, используйте команду git push:
```
git push <remote-name> <local-branch-name>
```
Например:
```
git push origin main
```
В целом, удаленный репозиторий является критически важным компонентом в процессе разработки с использованием Git, обеспечивая совместную работу, резервное копирование и управление доступом к коду.
## Совместная работа над проектом (fork, pull request)

### Как строится и для чего нужна совместная работа в системах контроля версий
Совместная работа в системах контроля версий (VCS) является критически важным аспектом современной разработкии других проектов, требующих управления и отслеживания изменений в файлах. 

#### Основные Принципы Совместной Работы
#### Использование Репозиториев
Совместная работа в VCS начинается с создания и использования репозиториев. Репозиторий может быть локальным (на вашем компьютере) или удаленным (на сервере, доступном через интернет). Удаленные репозитории позволяют нескольким разработчикам работать над одним проектом, обмениваясь изменениями через общий сервер.
#### Ветки и их Использование
Ветки являются мощным инструментом для параллельной работы над разными частями проекта. Разработчики могут создавать отдельные ветки для новых функций, исправления ошибок или экспериментальных изменений, не затрагивая основную линию кода. Это позволяет работать над несколькими задачами одновременно и затем сливать изменения в основную ветку после завершения и тестирования.
#### Коммиты и Слияние
Коммиты представляют собой точки сохранения изменений в репозитории. Разработчики делают коммиты в своей локальной ветке, а затем могут слить эти изменения с другими ветками или отправить их в удаленный репозиторий. Слияние (merge) позволяет объединить изменения из разных веток, решая конфликты, если они возникают.

#### Инструменты и Команды для Совместной Работы
##### Клонирование Репозитория
Чтобы начать работать над проектом, разработчики клонируют удаленный репозиторий на свой локальный компьютер:
```
git clone <remote-repo-url>
```
Это создает локальную копию репозитория, которая синхронизируется с удаленным.
##### Отправка и Получение Изменений
Разработчики отправляют свои локальные изменения в удаленный репозиторий с помощью команды git push:
```
git push <remote-name> <local-branch-name>
```
И получают изменения от других разработчиков с помощью команды git pull:
```
git pull <remote-name> <remote-branch-name>
```
Эти команды обеспечивают синхронизацию между локальными и удаленными репозиториями.
#### Преимущества Совместной Работы
##### Увеличение Продуктивности
Совместная работа в VCS позволяет нескольким разработчикам работать над одним проектом одновременно, что существенно увеличивает продуктивность. Каждый разработчик может работать над своей части проекта, не мешая другим.
##### Управление Доступом и Безопасность
VCS позволяет управлять доступом к коду, назначая разные уровни доступа для различных пользователей. Это обеспечивает безопасность и контроль над тем, кто может изменять код.
##### Резервное Копирование
Удаленный репозиторий служит как резервная копия проекта. Если локальная копия проекта будет утеряна или повреждена, ее всегда можно восстановить из удаленного репозитория.
##### Отслеживание Изменений
VCS позволяет отслеживать все изменения, внесенные в проект, включая информацию о том, кто сделал изменения, когда и почему. Это помогает в анализе первопричин возникновения ошибок и понимании структуры кода.
#### Пример Рабочего Процесса
##### Создание и Клонирование Репозитория
Разработчик создает новый репозиторий на GitHub или другом сервисе хостинга и клонирует его на свой локальный компьютер:
```
git clone https://github.com/user/repository.git
```
##### Создание Ветки и Работа Над Функцией
Разработчик создает новую ветку для реализации новой функции:
```
git checkout -b feature/new-feature
```
Он работает над этой функцией, делает коммиты и затем сливает изменения в основную ветку:
```
git checkout master
git merge feature/new-feature
```
##### Отправка Изменений в Удаленный Репозиторий
Разработчик отправляет свои изменения в удаленный репозиторий:
```
git push origin master
```
Этот процесс повторяется для каждого разработчика, работающего над проектом, что позволяет эффективно управлять кодом и сотрудничать в команде.
В целом, совместная работа в системах контроля версий является фундаментальным аспектом современной разработки, обеспечивая повышенную продуктивность, безопасность, управление доступом и резервное копирование кода.
### Инструкция по созданию pull request
Создание pull request (PR) является ключевым процессом в совместной работе над проектами в системах контроля версий, таких как Git, особенно на платформах như GitHub, GitLab или Bitbucket. 
##### Инструкция по созданию pull request:
1. Форкнуть Репозиторий (для внешних участников)

Если вы не имеете прямого доступа к репозиторию, вам необходимо форкнуть его. Форк создает копию репозитория в вашем аккаунте.
Зайдите в репозиторий, который вы хотите форкнуть.
Нажмите на кнопку "Fork" в правом верхнем углу страницы репозитория.
2. Склонировать Форкнутый Репозиторий

Склонируйте форкнутый репозиторий на ваш локальный компьютер.
```
git clone https://github.com/your-username/your-repo-name.git
```
3. Создать Новую Ветку

Создайте новую ветку для ваших изменений. Это помогает оставить основную ветку (master или main) чистой и стабильной.
```
git checkout -b feature/new-feature
```
или используя более современный синтаксис:
```
git switch -c feature/new-feature
```
4. Внести Изменения и Закоммитить

Внесите необходимые изменения в файлы, добавьте их в индекс и закоммитьте.
```
git add .
git commit -m "Description of your changes"
```
5. Отправить Изменения в Ваш Форкнутый Репозиторий

Отправьте ваши изменения в ваш форкнутый репозиторий.
```
git push origin feature/new-feature
```
6. Создать Pull Request

Перейдите на страницу вашего форкнутого репозитория на GitHub и создайте новый pull request.
Нажмите на вкладку "Pull requests" и затем на кнопку "New pull request".
Выберите базовый репозиторий и ветку, в которую вы хотите влить изменения (обычно master или main).
Выберите ветку, из которой будут вливаться изменения (вашу новую ветку).
Просмотрите изменения и убедитесь, что все корректно.
Нажмите на кнопку "Create pull request".

7. Описать Pull Request

Заполните форму pull request, включая заголовок и описание ваших изменений. Это поможет другим понять, что вы сделали и почему.
Введите заголовок и описание ваших изменений.
Добавьте любую дополнительную информацию, если необходимо (например, ссылки на задачи или комментарии).
Нажмите на кнопку "Create pull request".

8. Обсуждение и Code Review

После создания pull request, другие участники команды могут просмотреть ваши изменения, оставить комментарии и запросить изменения.
Рецензенты могут просматривать изменения на вкладке "Files changed".
Они могут оставлять комментарии и запросить изменения.
Вы можете вносить изменения и отправлять обновленные версии вашей ветки.

9. Слияние Pull Request

После того, как все изменения одобрены, владелец репозитория может слить pull request.
Перейдите на страницу pull request.
Проверьте, что все изменения корректны.
Нажмите на кнопку "Merge pull request" и затем на "Confirm merge".
Этот процесс обеспечивает прозрачность, позволяет进行 код-ревью и помогает поддерживать качество кода в проекте.

## Альтернативные системы контроля версий.
### Bitbucket
#### Описание
Bitbucket является популярной альтернативой GitHub, разработанной Atlassian. Она не только управляет кодом Git, но также предоставляет дополнительные функции для планирования проектов, совместной работы над кодом, тестирования и развертывания.
#### Преимущества
* Бесплатные неограниченные частные репозитории.
* Лучшая в своем классе интеграция с Trello и Jira.
* Поддержка создания, тестирования и развертывания с помощью интегрированного CI/CD.
### SourceForge
#### Описание
SourceForge является платформой для разработки и распространения проектов с открытым исходным кодом. Она поддерживает Git и Mercurial как системы контроля версий.
#### Преимущества
* Обширная всемирная зеркальная сеть.
* Интегрированное отслеживание проблем.
* Возможность просматривать код через браузер.
* Поддержка форумов, блогов и списков рассылки.
### Gogs
#### Описание
Gogs является легкой и быстрой selbsthosted альтернативой GitHub. Она позволяет независимо распространять двоичные файлы на различные платформы.
#### Преимущества
* Работает где угодно, включая недорогие устройства типа Raspberry Pi.
* Отличная производительность и эффективность.
* Минимальные требования к ресурсам.
### AWS CodeCommit
#### Описание
AWS CodeCommit является службой контроля версий на базе Git, предназначенной для совместной разработки программного обеспечения. Она интегрируется с другими сервисами AWS.
#### Преимущества
* Легкий доступ и интеграция с другими сервисами AWS.
* Без ограничений по размеру репозиториев.
* Поддержка HTTPS и SSH для передачи файлов.
### Beanstalk
#### Описание
Beanstalk является безопасным и мощным инструментом для управления репозиториями исходного кода. Он поддерживает Git и SVN.
#### Преимущества
* Поддержка хостинга Git и SVN.
* Быстрое и точное развертывание кода.
* Настройка конфигураций развертывания для каждой среды.
* Возможность оставлять два типа обратной связи: «Проблемы» и «Обсуждения».
### Allura
#### Описание
Allura является инструментом с открытым исходным кодом, размещенным на Apache. Она позволяет управлять отчетами об ошибках, репозиториями исходного кода, обсуждениями и т.д.
#### Преимущества
* Управление несколькими проектами.
* Функции управления исходным кодом для работы с Git, SVN и других систем.
* Mail-списки или форумы для открытых дискуссий.
* Вики для управления документацией.
### RhodeCode
#### Описание
RhodeCode является корпоративным инструментом с открытым исходным кодом для управления исходным кодом. Он поддерживает Git, Mercurial и Subversion.
#### Преимущества
* Проведение обзоров кода и улучшение качества кода.
* Автоматизация рабочих процессов.
* Управление разрешениями на уровне предприятия.
* Интеграция с корпоративными инструментами и премиум-поддержка в корпоративной версии.
### Mercurial
#### Описание
Mercurial является распределенной системой контроля версий, созданной как альтернатива Git. Она написана на Python и известна своей простотой и эффективностью.
#### Преимущества
* Децентрализованная архитектура.
* Быстрая и легкая в использовании.
* Используется в проектах seperti OpenOffice.org.
### TFVC (Team Foundation Version Control)
#### Описание
TFVC является централизованной системой управления версиями, используемой в Azure DevOps.
#### Преимущества
* Поддержка серверных рабочих областей и локальных рабочих областей.
* Масштабируемость до очень больших баз кода.
* Возможность аудита и отслеживания изменений.

Эти альтернативы предоставляют различные возможности и могут быть выбраны в зависимости от конкретных потребностей вашей команды или проекта.
