# Инструкция по работе с git

# Что это и для чего нужна система контроля версий?

Система контроля версий (Version Control System, VCS) – это инструмент, который используется для отслеживания изменений в файлах и документах, особенно в контексте совместной работы над проектами. Основные задачи такой системы:

1. **Отслеживание изменений**: Каждая версия файла сохраняется вместе с метаданными о том, когда и кем были внесены изменения. Это позволяет легко сравнивать разные версии файлов и видеть, какие именно правки были сделаны.
   
2. **Совместная работа**: В больших проектах часто работает несколько человек одновременно. Системы контроля версий позволяют нескольким разработчикам работать над одним проектом без риска перезаписи чужих изменений. Например, каждый может работать со своей копией проекта, а затем объединить свои изменения с основной версией.

3. **Возврат к предыдущей версии**: Если обнаруживается ошибка или нужно вернуться к более ранней версии кода, система контроля версий позволяет быстро восстановить предыдущие состояния проекта.

4. **Управление ветвлением**: Для реализации новых функций или исправления ошибок разработчики могут создавать отдельные ветки (branches). Ветвление позволяет безопасно экспериментировать с изменениями, не затрагивая основную версию проекта.

# Установка git и VSCode на ваш ПК.

Для установки Git и Visual Studio Code (VSCode) на Windows выполните следующие шаги:

Установка Git
Перейдите на официальный сайт Git: https://git-scm.com/download/win.
Скачайте установочный файл.
Запустите скачанный файл и следуйте инструкциям мастера установки.
После завершения установки проверьте работоспособность, открыв командную строку и введя команду git --version.

Установка Visual Studio Code
Зайдите на сайт VSCode: https://code.visualstudio.com/.
Нажмите кнопку «Download for Windows» и выберите нужную версию (64-bit или 32-bit).
Скачав установщик, запустите его и следуйте указаниям мастера установки.
По завершении установки запустите VSCode через меню «Пуск» или ярлык на рабочем столе.

#### Первая настройка git

После установки Git необходимо настроить глобальные параметры для вашего имени и электронной почты. Это важно, так как эти данные будут использоваться при каждом коммите.

1. Откройте терминал (командную строку).
2. Настройте имя пользователя:
   ```bash
   git config --global user.name "Ваше Имя"
   ```
3. Настройте электронную почту:
   ```bash
   git config --global user.email "ваш_адрес@example.com"
   ```
4. Проверьте настройки:
   ```bash
   git config --list
   ```

# Создание и базовая работа с локальным репозиторием.

Репозиторий — это хранилище данных, где хранятся все версии файлов проекта, история изменений и информация о них. Локальный репозиторий находится непосредственно на вашем компьютере.

### Инструкция по созданию локального репозитория

1. **Создайте новую папку для проекта**:
   ```bash
   mkdir my_project
   cd my_project
   ```

2. **Инициализируйте пустой репозиторий**:
   Команда `git init` создаст скрытый каталог `.git`, который будет содержать всю информацию о репозитории.
   ```bash
   git init
   ```

3. **Добавьте файлы в репозиторий**:
   Поместите необходимые файлы в папку проекта и добавьте их в индекс с помощью команды `git add`. Можно добавить сразу все файлы:
   ```bash
   git add .
   ```

4. **Зафиксируйте изменения (commit)**:
   Создайте первый коммит с описанием изменений:
   ```bash
   git commit -m "Initial commit"
   ```

### Базовая работа с локальным репозиторием

Вот основные команды для базовой работы с локальным репозиторием в Git:

1. **Проверка статуса репозитория**:
   ```bash
   git status
   ```

2. **Добавление файлов в индекс (staging area)**:
   Все новые или изменённые файлы должны быть добавлены в индекс перед фиксацией.
   ```bash
   # Добавляет все изменённые и новые файлы
   git add .

   # Или добавляет конкретный файл
   git add filename.txt
   ```

3. **Фиксация изменений (commit)**:
   Фиксирует текущие изменения с заданным сообщением.
   ```bash
   git commit -m "Описание изменений"
   ```

4. **История коммитов**:
   Просматривает список предыдущих коммитов.
   ```bash
   git log
   ```

5. **Откат изменений**:
   Возвращает файл к состоянию последнего коммита.
   ```bash
   # Отменяет изменения в файле, но оставляет его в рабочей директории
   git checkout -- filename.txt

   # Удаляет файл из индекса, но сохраняет изменения в рабочей копии
   git reset HEAD filename.txt
   ```

6. **Удаление файла**:
   Удаляет файл как из рабочего каталога, так и из индекса.
   ```bash
   rm filename.txt
   git rm filename.txt
   ```

# Ветки. Локальная работа с ветками в git.

Ветки (branches) в системе контроля версий, таких как Git, представляют собой параллельные линии развития проекта. Они позволяют разработчикам работать над разными функциями или задачами независимо друг от друга, не мешая основному потоку разработки.

### Зачем нужны ветки?

1. **Разделение задач**: Ветка создается для каждой новой функции или исправления ошибки. Это позволяет изолировать изменения и избежать путаницы в основном коде.
  
2. **Параллельная разработка**: Несколько разработчиков могут работать над различными частями проекта одновременно, используя разные ветки.

3. **Тестирование и стабилизация**: Новые функции тестируются в отдельных ветках, прежде чем интегрироваться в основную ветку. Это предотвращает внесение нестабильных изменений в рабочий проект.

4. **Экспериментирование**: Разработчики могут создавать временные ветки для экспериментов с новыми идеями, не рискуя повредить стабильный код.

Таким образом, ветки делают процесс разработки гибким и управляемым, позволяя команде эффективно сотрудничать и поддерживать качество кода.

### Базовая работа с ветками в git.

Вот основные команды для работы с ветками в Git:

1. **Создание ветки**:
   ```bash
   git branch new_branch
   ```

2. **Переключение на ветку**:
   ```bash
   git checkout existing_branch
   ```

3. **Объединение веток**:
   Сначала переключитесь на основную ветку, затем выполните слияние:
   ```bash
   # Переключаетесь на основную ветку
   git checkout main

   # Объединяете изменения из другой ветки
   git merge other_branch
   ```

4. **Удаление ветки**:
   После успешного слияния ветку можно удалить:
   ```bash
   # Удаляет локальную ветку
   git branch -d merged_branch
   ```

5. **Просмотр списка веток**:
   Показывает все локальные ветки:
   ```bash
   git branch
   ```

# Работа с удаленными репозиториями.

Удалённый репозиторий в Git - это копия репозитория, хранящаяся на сервере, которая позволяет нескольким разработчикам совместно работать над проектом. Он служит для синхронизации изменений между локальными репозиториями участников команды. Основные операции с удалённым репозитарием включают отправку изменений (push) и получение обновлений (pull). Использование удалённого репозитория позволяет разработчикам работать автономно, сохраняя возможность обмена изменениями и координации работы.

Работа с удалёнными репозиториями в Git включает взаимодействие с серверной частью системы контроля версий. Вот основные команды для этого:

1. **Клонирование репозитория**:
   Клонирует удалённый репозиторий на локальный компьютер:
   ```bash
   git clone https://github.com/user/repo.git
   ```

2. **Добавление удалённого репозитория (origin)**:
   Связывает локальный репозиторий с удалённым:
   ```bash
   git remote add origin https://github.com/user/repo.git
   ```

3. **Отправка изменений на удалённый сервер (push)**:
   Отправляет ваши коммиты на удалённую ветку:
   ```bash
   # Отправляет изменения текущей ветки на удаленную ветку
   git push origin your_branch_name
   ```

4. **Получение обновлений с сервера (pull)**:
   Загружает последние изменения с удалённой ветки и объединяет их с вашей локальной веткой:
   ```bash
   # Получает обновления и сливает их с текущей веткой
   git pull origin your_branch_name
   ```

5. **Просмотр списка удалённых репозиториев**:
   Показывает все подключённые удалённые репозитории:
   ```bash
   git remote -v
   ```

Эти команды позволят вам взаимодействовать с удалёнными серверами, такими как GitHub, GitLab или Bitbucket, для совместной работы над проектами.

# Совместная работа над проектом (fork, pull request)

Fork и Pull Request — это важные концепции для совместной работы над проектами в системах контроля версий, таких как GitHub.

### Fork

**Fork** — это создание копии (форк) чужого репозитория в вашем аккаунте. Это полезно, если вы хотите внести изменения в чужой проект, но не имеете права прямого редактирования оригинального репозитория.

1. На сайте GitHub перейдите в нужный репозиторий.
2. Нажмите кнопку "Fork".
3. Форкированный репозиторий появится в вашем профиле.

### Pull Request

**Pull Request (PR)** — это запрос на объединение ваших изменений с основным репозитарием. После внесения изменений в форкированном репозитории, вы создаёте PR, чтобы предложить владельцу основного репозитория принять ваши изменения.

1. Сделайте нужные изменения в своём форкированном репозитории.
2. Отправьте изменения на свой форк (push).
3. На странице своего форкированного репозитория нажмите "New pull request".
4. Укажите, какие изменения вы предлагаете, и оставьте комментарий.
5. Отправляйте PR.

Владелец основного репозитория сможет просмотреть ваши изменения, обсудить их и, возможно, принять ваш PR, объединив ваши изменения с основным проектом.

Эта схема позволяет безопасно вносить изменения в проекты, обеспечивая прозрачность процесса и возможность обсуждения предложений.

### Как строится и для чего нужна совместная работа в системах контроля версий

Совместная работа в системах контроля версий, таких как Git, необходима для эффективного взаимодействия между несколькими участниками проекта. Она позволяет распределять задачи, следить за изменениями и обеспечивать целостность кода. Рассмотрим ключевые аспекты совместной работы:

# Основные элементы совместной работы

1. **Централизованный репозиторий**: Все участники работают с одной общей базой кода, хранящейся на сервере. Это обеспечивает единый источник истины для всего проекта.

2. **Ветви (branches)**: Каждый участник может создать свою ветку для работы над конкретной задачей. Ветви помогают изолировать изменения и предотвращают смешивание разных частей кода.

3. **Коммиты (commits)**: Участники фиксируют свои изменения в виде коммитов, оставляя сообщения, объясняющие суть изменений. Коммиты создают историю проекта, которую можно просматривать и анализировать.

4. **Push и Pull**: Участники отправляют свои изменения на сервер (push) и получают обновления от других (pull). Это позволяет всем участникам иметь актуальную копию проекта.

5. **Merge и Conflict Resolution**: Когда две ветви сливаются (merge), возможны конфликты, если изменения касаются одних и тех же строк кода. Участники решают конфликты, выбирая правильные версии кода.

6. **Pull Requests (PR)**: Участник предлагает свои изменения другим членам команды через pull request. Другие участники могут просматривать, комментировать и одобрять предложенные изменения.

7. **Code Review**: Перед тем как изменения будут приняты в основную ветку, другие члены команды могут провести код-ревью, проверяя качество кода и соответствие стандартам проекта.

8. **Тегирование (tagging)**: Важные моменты в развитии проекта (например, выпуски версий) отмечаются тегами, что упрощает возврат к этим точкам в будущем.

### Преимущества совместной работы

1. **Эффективное распределение задач**: Различные участники могут работать над своими частями проекта одновременно, не мешая друг другу.

2. **Контроль качества**: Код-ревью и проверка изменений другими членами команды улучшают качество кода.

3. **Поддержание целостности проекта**: Благодаря централизованному репозиторию и строгому контролю изменений, проект остаётся целостным и предсказуемым.

4. **Простота возврата к предыдущим версиям**: История коммитов позволяет легко вернуться к любой точке в прошлом, если что-то пошло не так.

5. **Улучшение коммуникации**: Совместная работа стимулирует обсуждение и обмен знаниями внутри команды.

Совместная работа в системах контроля версий делает разработку программного обеспечения более организованной, безопасной и продуктивной.

### Инструкция по созданию pull request

Создание pull request (PR) — это важный этап совместной работы над проектом в системах контроля версий, таких как GitHub. Вот пошаговая инструкция по созданию pull request:

### Шаг 1: Подготовьте изменения

1. **Сделайте fork репозитория**. Если вы работаете над открытым проектом, сделайте форк репозитория в свой аккаунт на GitHub.
2. **Клонируйте форкированный репозиторий** на свой локальный компьютер:
   ```bash
   git clone https://github.com/your_username/forked_repo.git
   ```
3. **Создайте новую ветку** для своих изменений:
   ```bash
   git checkout -b feature-branch
   ```
4. **Внесите изменения** в код и зафиксируйте их:
   ```bash
   git add .
   git commit -m "Краткое описание изменений"
   ```
5. **Отправьте изменения** на ваш форк:
   ```bash
   git push origin feature-branch
   ```

### Шаг 2: Создайте pull request

1. Перейдите на страницу вашего форкированного репозитория на GitHub.
2. Найдите вкладку **"Compare & pull request"** или **"New pull request"**. Эта кнопка обычно расположена рядом с кнопкой "Branches".
3. Вы увидите сравнение между вашей веткой и основной веткой (обычно `main` или `master`) оригинального репозитория.
4. Проверьте, что всё правильно, и убедитесь, что ваша ветка указана слева, а основная ветка оригинального репозитория справа.
5. Напишите **заголовок** и **описание** для вашего pull request. Опишите, что было сделано и почему это важно.
6. При необходимости добавьте **метки**, **задания** или упомяните пользователей, которых вы хотите уведомить о вашем PR.
7. Нажмите кнопку **"Create pull request"**, чтобы отправить запрос.

### Шаг 3: Обсудите и улучшайте

1. Ожидайте **обратной связи** от владельцев проекта или других участников. Возможно, потребуется внести дополнительные изменения.
2. Если требуется, внесите **исправления** в вашу ветку:
   - Сделайте изменения локально.
   - Зафиксируйте их и отправьте на ваш форк.
   ```bash
   git commit -am "Исправлено по комментариям"
   git push origin feature-branch
   ```
3. Следите за **статусом проверки** (если применимо) и за отзывами других участников.

### Шаг 4: Закрытие pull request

1. Как только ваш pull request будет одобрен и все комментарии учтены, владелец проекта или администратор может **слить** (merge) ваши изменения в основную ветку.
2. После слияния вы можете **удалить** свою ветку, если она больше не нужна:
   - Локально:
     ```bash
     git checkout master
     git branch -D feature-branch
     ```
   - На удалённом сервере:
     ```bash
     git push origin --delete feature-branch
     ```

# Альтернативные системы контроля версий.

Хотя Git является самой популярной системой контроля версий, существуют и другие альтернативы, каждая из которых имеет свои особенности и области применения. Вот несколько известных альтернатив:

### 1. Mercurial (Hg)
Mercurial — это распределённая система контроля версий, схожая с Git. Она известна своей простотой и удобством использования. Некоторые пользователи предпочитают её за более интуитивный интерфейс и меньшее количество команд по сравнению с Git.

**Особенности:**
- Простота и удобство использования.
- Поддерживает крупные проекты и большие файлы.
- Хорошее сообщество поддержки.

**Применение:**
- Open Source проекты.
- Крупные корпоративные проекты.

### 2. Apache Subversion (SVN)
Subversion — одна из старейших систем контроля версий. Она использует централизованную модель, где существует центральный сервер, содержащий все версии файлов проекта.

**Особенности:**
- Централизованная архитектура.
- Хорошо подходит для крупных проектов с большим количеством файлов.
- Широко используется в корпоративных средах.

**Применение:**
- Корпоративные проекты с централизованным управлением.
- Проекты с большим числом участников и сложных процессов утверждения изменений.

### 3. Perforce Helix Core (P4D)
Perforce — это коммерческая система контроля версий с акцентом на производительность и масштабируемость. Она поддерживает управление большими бинарными файлами и хорошо подходит для игровых студий и компаний, работающих с мультимедийным контентом.

**Особенности:**
- Высокая производительность и поддержка больших файлов.
- Подходит для работы с графическим контентом и играми.
- Платформа для совместной работы и управления жизненным циклом продукта.

**Применение:**
- Игровые студии.
- Компании, работающие с мультимедиа и графическим дизайном.

### 4. Microsoft Team Foundation Server (TFS)/Azure DevOps
Microsoft TFS и Azure DevOps предлагают комплексные решения для управления жизненным циклом разработки ПО, включая контроль версий, непрерывную интеграцию и доставку.

**Особенности:**
- Интеграция с инструментами Microsoft.
- Комплексное решение для управления проектами.
- Удобство для разработчиков, использующих экосистему Microsoft.

**Применение:**
- Проекты, основанные на технологиях Microsoft.
- Корпорации, предпочитающие единую платформу для управления разработкой.

### 5. Bazaar (bzr)
Bazaar — это ещё одна распределённая система контроля версий. Её отличительной особенностью является простота интеграции с другими системами контроля версий и возможность работы в различных операционных системах.

**Особенности:**
- Гибкость и интеграция с другими системами.
- Легко адаптируется к различным рабочим процессам.

**Применение:**
- Открытые проекты с участием множества разработчиков.
- Проектные группы, которым важна гибкость и адаптивность инструментов.

Каждая из этих систем имеет свои сильные стороны и область применения. Выбор зависит от конкретных потребностей проекта, размера команды и предпочтений разработчиков.
